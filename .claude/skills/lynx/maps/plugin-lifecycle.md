# Plugin Lifecycle Map

## The 4 Stages

```
DECLARE → RESOLVE → LOAD → ACTIVATE
```

No plugin skips stages. No stage merges with another. A plugin's state in PluginRegistry
tracks exactly which stage it's in.

## DECLARE
- Source: all plugin.toml files in ~/.config/lynx/plugins/ and built-in plugins/
- Action: parse each plugin.toml into PluginManifest struct
- Validates: TOML is valid, required fields present, exports have no wildcards
- Failure: plugin is marked ERROR with parse failure — does NOT block other plugins
- Result: list of PluginManifest structs

## RESOLVE
- Source: all DECLARED manifests
- Action: build dep graph, topological sort, apply context filter
- Validates: no circular deps, binary deps exist on PATH
- Filters: plugins with disabled_in containing active context are EXCLUDED entirely
- Failure: circular dep = ERROR for all plugins in the cycle; missing binary = EXCLUDED with warning
- Result: two ordered lists — eager_load_order[], lazy_load_order[]

## LOAD
- Eager plugins: lx plugin exec <name> runs, shell evals output, functions/aliases defined
- Lazy plugins: wrapper functions emitted for each exported function (see maps/shell-flow.md)
- Validates: exported names match what plugin actually defines (lint — not hard block)
- Failure: plugin marks FAILED, lx doctor will report it, other plugins continue
- Result: PluginRegistry entries move to LOAD state

## ACTIVATE
- Action: register event subscriptions from manifest hooks[] field
- Each hook entry registers a handler with EventBus
- Failure: event registration failure marks plugin DEGRADED (loaded but hooks broken)
- Result: PluginRegistry entries move to ACTIVATE state

## State Machine

```
PENDING → DECLARE (parse) → DECLARED
DECLARED → RESOLVE (dep graph, context filter) → RESOLVED | EXCLUDED | ERROR
RESOLVED → LOAD (exec + eval) → LOADED | FAILED
LOADED → ACTIVATE (register hooks) → ACTIVE | DEGRADED
```

## What Lives Where

- DECLARE logic: lynx-manifest (parser) + lynx-plugin (discovery)
- RESOLVE logic: lynx-loader (dep graph, topo sort, context filter)
- LOAD logic: lynx-plugin/lifecycle.rs + lx plugin exec command
- ACTIVATE logic: lynx-plugin/lifecycle.rs + lynx-events (subscription)
- State tracking: PluginRegistry in lynx-plugin

## Lazy Plugin Trigger

When a lazy plugin's wrapper function is called:
1. Wrapper calls `lx plugin exec <name>`
2. Output evaled — moves plugin from RESOLVED to LOADED to ACTIVE in one call
3. Real functions now defined — wrapper unregisters itself
4. Original command re-invoked

The wrapper is generated by `lynx-plugin/src/lazy.rs` (`generate_lazy_wrappers`) during lx init output.
